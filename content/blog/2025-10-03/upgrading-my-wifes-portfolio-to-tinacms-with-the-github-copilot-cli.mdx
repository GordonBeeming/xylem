Not long ago, I wrote about how I finally built my wife, [Tiani Beeming](https://tianibeeming.com)'s, portfolio website in about five hours after a 10-year delay, all thanks to an AI-driven workflow. The site looked great, and it was a huge win.

But there's always a "but", isn't there?

While the site was live, updating the content — adding new portfolio pieces, tweaking text — was still a manual process for me. Pull down the code, make the change, redeploy. The goal was always to let Tiani manage her own content without needing me as the middleman. That's where [TinaCMS](https://tina.io/) comes in, a Git-based headless CMS that seemed like a good fit.

## The plan: a few hours of manual work

As I've mentioned before, I'm a backend dev, not a UI guy. So my first step was to figure out what it would actually take to integrate TinaCMS manually. I had a quick chat with [Jack Pettit](https://au.linkedin.com/in/jackdevau) from the TinaCMS team to get a realistic estimate.

He figured a couple of hours, maybe 5 or so, to get everything installed, configured, and hooked up to the components. Totally doable, but still a decent chunk of an evening.

## The experiment: "Hey Copilot, can you do this?"

While playing with the new **GitHub Copilot CLI**, a thought occurred to me. Instead of following the docs myself, what if I just asked Copilot to do it for me?

I opened my terminal and typed a single prompt. No step-by-step plan, just the end goal.

```text title="Copilot CLI prompt"
I'd like to completely convert this project to use tina.io cms...
please do the complete conversion for me...
any json files in the data folder should be able to be managed using the
tina interfaces as normal content
```

## The AI takes the wheel

The Copilot CLI didn't come back with a list of instructions for me to follow. It just started *working*.

It suggested shell commands one by one. My role shifted from "developer" to "supervisor" — it would propose a command, I'd review it and approve or skip.

What I didn't expect was how well the safety model worked. For a read-only command like `grep`, Copilot gives you the option to approve just once or trust it for the entire session. But for anything destructive — deleting files, wiping `node_modules` — that "approve for session" option disappears. It forces a specific yes/no every single time. I was always in control, I just wasn't doing any of the typing.

## The result: a five-hour job done in minutes

I was expecting friction. I thought I'd have to intervene, correct a mistake, or at least guide it past something. I didn't. It just worked.

In a fraction of the time I had budgeted, the entire site was converted to TinaCMS. Packages installed, schema defined from my JSON files, React components wrapped for visual editing.

The part that genuinely surprised me was the documentation. I expected maybe an updated `README.md`. Instead it generated a whole suite of files, including a detailed `TINA_CMS_GUIDE.md`, and it correctly placed the supporting docs in my project's custom `/docs` and `/tasks` folders.

That wasn't luck. In a previous session I'd asked a Copilot agent to update the repository's `copilot` instructions file with those conventions. The CLI read that file and followed the rules I'd already set out. It's a good reminder that the instructions you write for your repo have real teeth.

## Conclusion: The New Standard for Complex Tasks

This experience solidified the "developer as orchestrator" mindset for me. The GitHub Copilot CLI isn't just a tool for asking questions; it's an active agent capable of executing complex, multi-step refactoring tasks on your behalf.

I still need to do some final checks and get the updated site deployed this weekend, but this workflow is a game-changer. The big, daunting tasks on the backlog suddenly seem a lot less scary. I'm excited to see what else I can throw at it. 🚀
