We've all been there. You have a simple goal that should take 15 minutes, but hours later you're deep in network troubleshooting with no end in sight. My goal was straightforward: securely expose a development mail-catcher tool, `smtp4dev`, running in an Azure Container App Environment to the internet using a Cloudflare Tunnel.

It's a good pattern for giving distributed teams access to internal tools without dealing with VPNs or public IP addresses. But this time, it just wouldn't work.

## The Setup

The architecture has three parts:
1.  **A "Mail Catcher" Container App (`mail-catcher-app`):** Running the `smtp4dev` image, this app accepts SMTP traffic on a TCP port and exposes a web UI.
2.  **A "Tunnel Agent" Container App (`tunnel-agent-app`):** Running the `cloudflared` agent in a separate container within the same environment.
3.  **A Cloudflare Tunnel:** Configured with two public hostnames, one for the HTTP web UI and one for the TCP SMTP traffic.

HTTP worked straight away. TCP? Nothing. Time to start eliminating variables.

## Step 1: Check the obvious - ports and ingress

My first suspect was a port mismatch. In Azure Container Apps, to expose both HTTP and TCP traffic, you configure the main ingress for HTTP and then add an "Additional TCP port". I set the main ingress for HTTP on port 80 and added TCP port `2525`.

I double-checked the Cloudflare Tunnel config was pointing the TCP service at port `2525`.

Still nothing. Connection timed out every time.

## Step 2: Can the containers even talk to each other?

Is this a Cloudflare problem or an internal Azure network problem? The only way to know is to test from inside the environment. I dropped a shell into `tunnel-agent-app` to check if it could reach `mail-catcher-app` directly.

My containers are locked down with no `telnet` or `nc`. `bash` has a built-in TCP test that works fine as a substitute:

```bash title="Terminal"
# This command attempts to open a TCP connection to the specified host and port.
# It's a great alternative when telnet/nc are not available.

(echo > /dev/tcp/HOST/PORT) &> /dev/null && echo "Success" || echo "Failure"
````

I ran the test against both the HTTP port and the SMTP port:

```bash title="Terminal"
# Test the HTTP port (web UI)
$ (echo > /dev/tcp/mail-catcher-app/80) &> /dev/null && echo "Success" || echo "Failure"
Success

# Test the TCP port (SMTP)
$ (echo > /dev/tcp/mail-catcher-app/2525) &> /dev/null && echo "Success" || echo "Failure"
Failure
```

Port 80 succeeded. Port 2525 failed. That narrowed it down considerably - the containers could see each other on the network, but `mail-catcher-app` wasn't accepting connections on the SMTP port from other containers.

### Step 3: A confusing contradiction

Here's where it got weird. I had two facts that couldn't both be true at the same time.

What was correctly configured:

* **Azure Ingress:** The `mail-catcher-app` had the "Additional TCP port" set up and enabled for `2525`. The Azure Portal showed everything in order.
* **Application Logs:** The logs inside `mail-catcher-app` said `SMTP Server is listening on port 2525 (::)`. The app was up, listening on the right port, on all interfaces.

And then the thing that contradicted all of it:

* **Network Test:** The `bash` test from `tunnel-agent-app` to `mail-catcher-app` on port 2525 still timed out.

App listening: yes. Network path configured: yes. Connection working: no. When you've got a running application that says it's listening but nothing on the same internal network can reach it, the problem is usually somewhere in the path between the two, not in the app itself.

### The final twist: it's always DNS

The only variable I hadn't changed was the hostname. And I almost didn't touch it, because the Azure FQDN was already working fine for the HTTP web UI:

`mail-catcher-app.internal.region.azurecontainerapps.io`

My `cloudflared` agent was connecting to `http://mail-catcher-app.internal.region...io` without any issue. So why would the same hostname work for HTTP on port 80 but fail for TCP on port 2525?

That was the red herring I'd been chasing. On a hunch, I switched to the short internal service name in my `bash` test:

```bash title="Terminal"
# Test the TCP port with the SHORT name
$ (echo > /dev/tcp/mail-catcher-app/2525) &> /dev/null && echo "Success" || echo "Failure"
Success
```

It worked. Instantly.

That was the key. The long FQDN resolves to the **public edge** of the Container App Environment. This managed ingress is responsible for handling the main HTTP traffic but has no knowledge of the **internal-only "Additional TCP ports"**. When the `cloudflared` agent tried to connect to the FQDN on port 2525, the traffic was essentially hitting a wall that didn't have that port open.

The short name, `mail-catcher-app`, uses the built-in DNS service within the Container Apps environment, which resolves directly to the container's private, internal IP, bypassing the public ingress and allowing access to the internal TCP port.

I updated the service URL in my Cloudflare Tunnel configuration to use the short name:
`https://mail-catcher-app:2525`

Everything lit up. The tunnel connected, and SMTP traffic started flowing. After all that, the solution was just to use the right name for the right job.

As always, it was DNS.
