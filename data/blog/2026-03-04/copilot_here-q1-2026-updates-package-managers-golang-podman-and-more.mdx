## 1. Install Anywhere: Package Manager Support

The biggest user-facing change this quarter is that `copilot_here` is now available through the package managers you already use. No more piping scripts from the internet — just install it the way you install everything else.

**.NET Global Tool:**
```bash
dotnet tool install -g copilot_here
```

**Homebrew (macOS):**
```bash
brew tap gordonbeeming/tap
brew install --cask copilot-here
```

**Homebrew (Linux):**
```bash
brew tap gordonbeeming/tap
brew install copilot_here
```

**WinGet (Windows):**
```powershell
winget install GordonBeeming.CopilotHere
```

All three channels are automatically published on every release via CI, so you'll always get the latest version through your package manager's normal update flow.

The existing `curl`/`iwr` install scripts continue to work if you prefer that approach.

## 2. Golang Image Variant

We've added a new `golang` image variant with the full Go toolchain pre-installed, including `go` and `gofmt`.

Use it with the `--golang` or `-go` flag:

```bash
copilot_here --golang "add unit tests for this Go module"
copilot_yolo --golang "refactor this handler"
```

This brings our image lineup to: **Base**, **.NET** (8/9/10), **Playwright**, **Rust**, and now **Golang** — plus compound variants like `dotnet-playwright` and `dotnet-rust`.

## 3. Container Runtime Freedom

Up until now, `copilot_here` required Docker. That's no longer the case.

We now support **Podman** and **OrbStack** as container runtimes, with automatic detection:

- **Podman**: Full support including rootless containers. Auto-detects Podman Compose support.
- **OrbStack**: Automatically detected when your Docker context is set to OrbStack.
- **Auto-detection**: Tries Docker first, then falls back to Podman.

You can also explicitly configure your preferred runtime:

```bash
# See what runtime is currently in use
copilot_here --show-runtime

# List all available runtimes on your system
copilot_here --list-runtimes

# Set your preferred runtime (per-project or global)
copilot_here --set-runtime podman
copilot_here --set-runtime-global docker
```

Thanks to [@tudorsmt](https://github.com/tudorsmt) for the original request in [#36](https://github.com/GordonBeeming/copilot_here/issues/36).

## 4. Model Configuration

You can now configure which AI model `copilot_here` uses, without passing `--model` every time.

```bash
# Set model for the current project
copilot_here --set-model claude-sonnet-4.5

# Set model globally for all projects
copilot_here --set-model-global gpt-5

# Override saved model for one session
copilot_here --model gpt-5-mini

# View current configuration
copilot_here --show-model

# List available models
copilot_here --list-models
```

**Configuration priority:** CLI argument > local config > global config > default.

A special `default` value lets you override a global setting at the local level to explicitly use Copilot CLI's default model.

## 5. Custom Mount Paths

Previously, when you mounted a directory, the container path was always auto-computed to match the host path. Now you can specify both sides using the `host:container` format:

```bash
# Mount with a custom container path
copilot_here --mount /host/data:/app/data

# Works with read-write mounts too
copilot_here --mount-rw /host/output:/app/output
```

This is especially useful on Windows where host paths don't map cleanly to Linux container paths, or when your project expects files at a specific location inside the container.

Thanks to [@Tasteful](https://github.com/Tasteful) for the request in [#62](https://github.com/GordonBeeming/copilot_here/issues/62).

## 6. Enhanced Image Name Handling

You can now use absolute image references with `copilot_here`, making it possible to use custom Docker images that aren't hosted under the default `ghcr.io/gordonbeeming/copilot_here` registry.

This was a frequently requested feature — thanks to [@christiansparre](https://github.com/christiansparre) for raising it in [#68](https://github.com/GordonBeeming/copilot_here/issues/68).
